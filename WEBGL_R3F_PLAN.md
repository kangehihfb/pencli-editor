# WebGL/R3F 편집 캔버스 구현 계획

## 목적

현재 목표는 펜 드로잉, 이미지, 텍스트, 입력 요소를 하나의 WebGL canvas 기반 구조 안에서 안정적으로 다루는 편집 캔버스 구조를 검증하는 것이다.

WebGL canvas, DOM canvas, 일반 DOM 요소가 각각 다른 기준으로 움직이면 줌/팬, 좌표 동기화, 레이어 우선순위, 입력 이벤트 충돌이 계속 발생할 수 있다. 따라서 가능한 요소는 하나의 WebGL canvas 기준으로 모으고, 실제 DOM 상호작용이 필요한 요소만 `drei/Html`을 통해 제한적으로 사용한다.

## 구현 우선순위

### 1. WebGL canvas 기반 구조 확정

- 모든 요소의 위치, 크기, 줌, 팬 기준을 하나의 WebGL canvas 기준으로 통일한다.
- 화면 이동과 확대/축소가 WebGL 카메라 또는 씬 기준으로 일관되게 동작하도록 구조를 잡는다.
- 이후 드로잉, 이미지, 텍스트, DOM 요소가 같은 좌표계를 공유할 수 있는지 확인한다.

완료 기준:

- 줌/팬 후에도 오브젝트의 위치가 의도한 좌표에 유지된다.
- WebGL 오브젝트와 `drei/Html` 요소가 같은 기준으로 배치된다.
- 좌표 변환 로직을 별도 기준 없이 한 곳에서 설명할 수 있다.

### 2. WebGL mesh 기반 드로잉 PoC

- 펜 필기를 `react-three-fiber`의 `mesh` 기반 WebGL 오브젝트로 구현할 수 있는지 실험한다.
- 우선 `mesh` 기반 구현을 중심으로 보고, 필요하면 `line`, `tube`, `points` 방식도 비교한다.
- 드로잉 결과가 이미지, 텍스트, DOM 요소와 같은 좌표계에서 움직이는지 확인한다.

완료 기준:

- 마우스 또는 펜 입력으로 WebGL canvas 내부에 선을 그릴 수 있다.
- 줌/팬 이후에도 드로잉 위치가 틀어지지 않는다.
- 최소한의 선 굵기, 색상, 지우기 또는 초기화 기능을 검증한다.

### 3. 이미지/텍스트 요소 WebGL 내부 배치 검증

- 이미지는 texture가 적용된 plane mesh 방식과 `drei/Html` 방식 중 적합한 방향을 비교한다.
- 텍스트는 `drei/Text`, texture, `drei/Html` 중 사용 목적에 맞는 방식을 검토한다.
- 이미지와 텍스트가 드로잉 레이어와 함께 자연스럽게 선택, 이동, 확대/축소되는지 확인한다.

완료 기준:

- 이미지와 텍스트를 WebGL 좌표 기준으로 배치할 수 있다.
- 드로잉과 이미지/텍스트의 상대 위치가 줌/팬 후에도 유지된다.
- WebGL 렌더링 방식과 `drei/Html` 방식의 장단점을 정리한다.

### 4. DOM 상호작용 요소 검증

- `input`, `textarea`, 버튼 같은 실제 DOM 상호작용 요소를 `drei/Html`로 배치한다.
- 포커스, 키보드 입력, 클릭, 드래그가 WebGL 줌/팬 및 선택 이벤트와 충돌하지 않는지 확인한다.
- DOM 요소를 언제 WebGL 내부 HTML로 둘지, 언제 WebGL 오브젝트로 대체할지 기준을 만든다.

완료 기준:

- `drei/Html` 내부 입력 요소에 정상적으로 포커스하고 값을 입력할 수 있다.
- 입력 중에는 캔버스 드래그/단축키와 충돌하지 않는다.
- DOM 요소와 WebGL 오브젝트의 이벤트 우선순위 규칙을 설명할 수 있다.

### 5. 레이어/선택/이벤트 체계 정리

- `z`축, raycaster, pointer event 기준으로 어떤 요소가 선택되고 상호작용되는지 규칙을 정리한다.
- 드로잉, 이미지, 텍스트, DOM 요소가 겹칠 때 입력 우선순위를 검증한다.
- 선택, 이동, 드래그, 편집 모드가 서로 충돌하지 않도록 상태 구조를 잡는다.

완료 기준:

- 겹친 요소 중 어떤 요소가 클릭되는지 예측 가능하다.
- 선택된 요소만 이동/편집된다.
- 드로잉 모드와 선택 모드가 명확히 분리된다.

### 6. 성능 측정 환경 추가

- `r3f-perf`를 추가해 FPS, 렌더링 비용, 오브젝트 증가에 따른 성능 변화를 확인한다.
- 드로잉 포인트 수, 이미지 수, `drei/Html` 요소 수가 늘어날 때 병목을 관찰한다.
- 성능 기준을 바탕으로 WebGL 오브젝트와 DOM HTML 사용 범위를 조정한다.

완료 기준:

- 개발 화면에서 기본 성능 지표를 확인할 수 있다.
- 드로잉/이미지/HTML 요소 수를 늘리며 성능 변화를 관찰할 수 있다.
- 병목이 발생하는 요소 유형을 기록한다.

### 7. 디버그/데모 조작 패널 추가

- `leva`를 추가해 좌표, z값, 줌, 오브젝트 속성, 테스트 모드를 실시간으로 조절한다.
- 데일리 또는 리뷰에서 현재 구조를 빠르게 설명하고 검증할 수 있는 데모 상태를 만든다.
- 구현 중 좌표 틀어짐, 레이어 충돌, 이벤트 충돌을 빠르게 재현할 수 있게 한다.

완료 기준:

- 주요 오브젝트의 위치, z값, 색상, 크기 등을 패널에서 조절할 수 있다.
- 드로잉/선택/DOM 입력 테스트 모드를 전환할 수 있다.
- 리뷰 시 동일한 시나리오를 반복해서 보여줄 수 있다.

## 먼저 구현할 범위

첫 번째 구현은 완성형 편집기가 아니라 편집기 구조 검증용 데모로 시작한다.

포함할 기능:

- WebGL 캔버스 기본 씬
- 줌/팬 가능한 카메라 또는 컨트롤
- WebGL mesh 기반 간단한 펜 드로잉
- 이미지, 텍스트, input 요소 추가
- 이미지, 텍스트, input, 드로잉 선 선택
- 선택한 요소 개별 이동
- 이미지와 텍스트는 WebGL 오브젝트로 배치
- input은 `drei/Html` 기반 DOM 요소로 배치
- raycaster/pointer event 기반 선택 테스트

이 범위가 안정화되면 펜 드로잉, 이미지, 텍스트, DOM 입력 요소를 하나의 WebGL canvas 기반 구조에서 계속 가져갈 수 있을지 판단한다.

## 주요 확인 질문

- 드로잉 결과는 편집 가능한 벡터 오브젝트여야 하는가, 아니면 렌더링 결과만 유지해도 되는가?
- 텍스트는 실제 입력/편집이 필요한가, 아니면 표시용 텍스트가 대부분인가?
- 이미지와 텍스트는 자유롭게 이동/리사이즈되어야 하는가?
- DOM 요소는 편집 모드에서만 필요할지, 항상 화면에 보여야 할지 정해야 한다.
- 저장 데이터 구조는 WebGL 좌표 기준으로 설계할지 별도 문서 좌표계를 둘지 결정해야 한다.

## 참고 라이브러리

- `react-three-fiber`: React 기반 Three.js 씬 구성
- `drei`: Html, Text, controls 등 R3F 보조 유틸
- `three.js`: raycaster, geometry, material 등 WebGL 기반 기능
- `r3f-perf`: R3F 성능 측정
- `leva`: 데모 및 디버그 조작 패널
